2. 일반적인 방법 : 한 회사에서 3개의 서비스를 오픈
   - 이른 아침 : A 서비스의 접속량이 높음
     + 이 접속량을 받쳐주기 위해 3대의 서버에 해당하는 자원이 필여
     + B와 C는 이 시간대 접속량이 낮지만, 정오가 되면, B 서비스만 접속량이 많아짐
     + 밤이 되면 C 서비스 접속량이 많아짐
     + A 서비스의 경우, 다른 시간대 트래픽이 적더라도, 아침 시간에 서비스 운영을 위해서 어쩔 수 없이 3대의 서버 할당
   - 따라서, 총 9대의 서버가 필요 
   - 기존 시스템의 경우 서버 장애 상황에 대비해 한 대의 여분 서버가 필요 : 각 서비스별 백업 서버를 둬서 운영 중인 서버가 죽었을 때, 즉각적인 대응을 할 수 있도록 조치 (3대 서버 추가 필요)
   - 서비스 중단이 허용되는 경우, 모든 서버를 내렸다가 업데이트 작업 후 다시 올림
     + 무중단 서비스를 해야 하는 경우, 한 서버씩 내렸다가 업데이트 작업 후 서비스를 올리는 방식으로 진행

3. Kubernetes 사용
   - 하룻동안 A, B, C의 서비스 평균 트래픽을 계산하면 4대분의 자원 필요
   - Auto-Scaling 기능이 있으므로 새벽, 정오, 저녁 상관없이 트래픽 양에 따라 서비스의 자원량을 변경시켜줌
   - 장애가 난 서버 위에 있는 서비스들이 다른 서버로 자동으로 옮겨주는 Auto-Healing 기능이 있으므로, 한 서버에 장애가 나더라도 유지
   - Deploy라는 Object를 통해 업데이트 방식에 대해 업데이트 방식에 대해 자동적 처리
   - 여러 기능들에 대해 운영 자동화 제공
